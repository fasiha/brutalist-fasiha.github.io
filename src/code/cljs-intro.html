Some terminology

There are many dichotomies in the world, and for webapps there exists a divide between the client (or front-end) and the server (or back-end). A client is a computer program like a web browser (Firefox, Safari, Chrome, that lot) that requests some assets, identified by URLs, from a server, which is another program running on potentially a different computer connected to the first through a network.

On the server, a second dichotomy exists between static content versus dynamic content. If you remember copying files to GeoCities‚Äô FTP, or to ~/public_html/ on your university shell account, those files were being served as static content. That is, a web server was converting URLs to file paths, and sending those files to visitors. Each visitor to a given URL saw the same content.

So if static web content maps web requests to files, then dynamic content involves running code to customize each response to the same URL. The server could be doing anything imaginable. Those visitor counters you used to see on people‚Äôs websites were generated dynamically: the page itself might be served statically, but it would include an image which was custom-generated for each visitor, incrementing the counter.

There‚Äôs a whole class of web functionality that requires dynamic generation on the server~~~usually anything involving coordination between visitors or persisting some data to permanent memory. However, it is also amazing how much can be done with a static website, and there are large classes of webapps that need no dynamic content.

Personally, I wrote client-side-only (entirely static) webapps for years before my desire for coordination and persistance forced me to revisit dynamic web servers. This was because, first, static websites are cheap to host (GitHub will host it for free, up to 100 GB traffic per month), and second, they present a microscopic attack surface for malicious evildoers. This tutorial is light on the front-end side of the house: if that is your main interest, go read about re-frame [GitHub]: I use it for all my webapps.
